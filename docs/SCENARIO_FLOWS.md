# AI 客服助手 - 场景流程设计

## 场景一：KYB/KYC 资料完整性检查

### 用户问法示例
- "为什么我的企业认证没过？"
- "KYB 还差什么资料？"
- "我已经提交了怎么还没通过？"
- "公司认证下一步做什么？"

### Agent 流程

```
用户输入 → 意图识别（KYB问题）
  → 查询用户 KYB 状态（get_kyb_status）
  → 识别缺失项 / 退回项 / 审核中项
  → 调用规则知识解释每项要求
  → 输出：当前状态 + 缺项说明 + 每项要求 + 建议动作
  → 如需人工介入，输出工单摘要
```

### 回复结构

```
1. 结论
   你的企业认证当前状态：[审核中 / 材料缺失 / 已退回]

2. 依据
   - [已提交项及状态]
   - [缺失/不合规项及原因]

3. 下一步
   - [缺失项的补交流程]
   - [每项的具体要求]
   - [预计审核时长]

4. 工单摘要（如需）
   - 类型：KYB/资料补充
   - 当前状态：[...]
   - 缺失材料：[...]
```

### 槽位要求
- 用户 UID（从上下文补全，或追问）
- 认证单号（如有多笔，追问是哪一笔）

## 场景二：提币未到账排查

### 用户问法示例
- "我昨晚提的 USDT 还没到账"
- "提币一直 pending"
- "链上成功了为什么平台没到账"
- "我提现卡住了"

### Agent 流程

```
用户输入 → 意图识别（提现异常）
  → 查询最近提现单（get_withdraw_status）
  → 若有多笔：最小追问（"你最近有 3 笔，我看到了...你要查哪笔？"）
  → 查平台提现状态（internal_status）
  → 查链上状态（txid + confirmations）
  → 查钱包/风控状态
  → 按规则归因
  → 输出诊断 + 建议
  → 如需人工，输出工单摘要
```

### 诊断归因（规则引擎）

| 平台状态 | 链上状态 | 归因 |
|---------|---------|------|
| 审核中 | 无 TXID | 平台风控审核，请等待 |
| 已广播 | 0 confirmations | 链上广播中，等待确认 |
| 已广播 | N confirmations（未达要求）| 确认数不足，继续等待 |
| 已出币 | 有 TXID，链上确认足够 | 对方平台入账中 |
| - | TXID 在对方平台查不到 | 收款地址/网络错误 |
| 审核拒绝 | - | 风控拦截，请查看原因或提交工单 |
| 钱包维护 | - | 对方钱包维护中，查看公告 |

### 回复结构

```
1. 结论
   你这笔 [币种] 提现（[网络]）：
   当前平台状态：[平台状态]
   当前链上状态：[确认数/N 确认]

2. 依据
   - 平台状态说明 [...]
   - 链上状态说明 [...]

3. 下一步
   - [当前建议]
   - [何时需要提交工单]

4. 工单摘要（如需）
   - 类型：提现未到
   - 币种/网络：[...]
   - 金额：[...]
   - TXID：[...]
   - 状态：[...]
```

### 槽位要求
- 币种（BTC/USDT/ETH...）
- 网络（TRC20/ERC20/Arbitrum...）
- 金额（可选）
- TXID（可选，有则更准）
- 时间（可选）

## 场景三：充值未到账排查

### 用户问法示例
- "我充了 USDT 但没到账"
- "为什么充值没入账"
- "我转过去了但是零"

### Agent 流程

```
用户输入 → 意图识别（充值未到）
  → 引导用户提供：币种 + 网络 + 金额 + TXID（如无则尝试从上下文推断）
  → 查询链上状态（txid + confirmations）
  → 查询充值状态（credit_status + required_confirmations）
  → 查钱包/网络状态（是否维护）
  → 按规则归因
  → 输出诊断 + 建议
  → 如需，输出补单工单
```

### 诊断归因

| 充值状态 | 原因 |
|---------|------|
| 确认数不足 | 链上确认中，等待达到 [N] 确认 |
| Memo 缺失/错误 | 对方平台需要 Memo，你的转账疑似缺失，建议提交工单 |
| 地址不匹配 | 充值地址与平台不匹配，建议提交工单 |
| 金额低于最小充值量 | [币种] 最小充值量为 [X]，你的金额为 [Y] |
| 钱包维护 | [币种/网络] 当前维护中，查看公告 |
| 已入账 | 已到账，请查看账户资产 |

### 回复结构

```
1. 结论
   你这笔 [币种] 充值（[网络]）：
   当前状态：[未入账原因]

2. 依据
   - 链上确认数：[N] / [要求]
   - 平台充值状态：[...]

3. 下一步
   - [建议动作]

4. 工单摘要（如需）
```

### 槽位要求
- 币种 + 网络（必填）
- 金额（可选）
- TXID（可选，有则精确定位）

## 场景四：钱包/网络状态查询

### 用户问法示例
- "ATOM 现在能提现吗？"
- "POL 能充值吗？"
- "为什么我不能提这个网络？"
- "这个链在维护吗？"

### Agent 流程

```
用户输入 → 意图识别（网络状态查询）
  → 查询钱包状态（get_wallet_network_status）
  → 查询是否有维护公告
  → 如有替代网络，一并告知
  → 输出状态 + 影响 + 建议
```

### 回复结构

```
1. 结论
   [币种] [网络]：
   充值：[支持/暂停]
   提现：[支持/暂停]

2. 依据
   - 当前状态 [...]
   - 公告：[如有]

3. 下一步
   - [替代网络建议，如有]
   - [预计恢复时间，如有维护]
```

## 场景五：知识问答（通用）

### 用户问法示例
- "什么是 Memo/Tag？"
- "TRC20 和 ERC20 有什么区别？"
- "提币手续费多少？"
- "什么是 KYC？"

### Agent 流程

```
用户输入 → 意图识别（知识问题）
  → 判断是否涉及特定币种/网络
  → 检索知识库（helpcenter / rules / FAQ）
  → 结合用户上下文（如有）生成回答
  → 先结论，后解释，附来源
```

### 回复结构

```
1. 结论
   [一句话直接回答]

2. 依据
   [结合用户场景的知识解释，附来源链接]

3. 下一步（如相关）
   [如有相关操作建议]
```
